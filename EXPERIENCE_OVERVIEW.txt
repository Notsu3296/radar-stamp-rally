レーダー・スタンプラリー 体験概要
==================================================

このメモは、GPTと一緒に整理した「体験の核」と、その体験を実現するための基本仕様案です。

本企画は、単なる「地図アプリ」ではありません。
体験の中心は、探知して、近づいて、現地で意味が開くポイントラリーです。

そのため、仕様の中心は正確なナビゲーションではなく、迷いすぎない探索感に置きます。


1. 体験コンセプト
--------------------------------------------------

利用者は、目的地名や説明を事前に知らされません。
代わりに、現在地から見た「反応」の方向だけを頼りに歩きます。

目的地に近づくと反応が強まり、到達範囲に入るとチェックインできます。
チェックインすると、その地点の意味や解説が初めて表示されます。

体験の基本的な流れ:

  探知する
  → 方向を見る
  → 歩く
  → 近づく
  → チェックインする
  → 意味が分かる

この流れを崩さないことが、体験設計上もっとも重要です。


2. 基本画面
--------------------------------------------------

画面はできるだけ少なくします。
主画面に必要な要素は、以下の程度で十分です。

  探知ボタン
    現在地と方位を取得し、目的地の反応を表示する。

  レーダー
    目的地の方向と近さを示す。

  反応リスト
    到達圏内のチェックイン候補を表示する。

  履歴
    訪問済み地点と獲得内容を見る。

  メニュー
    QR読み取り、設定、説明、リセットなどを置く。

地図は基本的に不要です。
地図を出すと「答え」が見えてしまい、探索体験が弱くなります。


3. レーダー仕様
--------------------------------------------------

今回の体験では、レーダーは常時表示ではなく、探知ボタンを押したときだけ一時表示する方式が合っています。

探知の動き:

  1. 利用者が立ち止まる。
  2. スマホの上端を進みたい方向へ向ける。
  3. 「探知」を押す。
  4. その時点の現在地と端末方位を取得する。
  5. ドットを10秒表示する。
  6. 残り5秒でドットを薄くする。
  7. 10秒後にドットを消す。
  8. もう一度押すと再探知する。

この方式なら、スマホを見ながら歩き続けにくくなります。
安全面でも説明しやすい仕様です。


4. 方位の扱い
--------------------------------------------------

方位は、探知時の値で固定します。

重要なのは、表示中に端末を回してもドットを回さないことです。
端末方位センサーは時間でズレることがあるため、常時追従させると体験が不安定になります。

推奨仕様:

  ・探知時に現在地を取得する。
  ・探知時に端末方位を取得する。
  ・その方位を固定値として保存する。
  ・10秒間だけ、その固定値でドットを表示する。
  ・表示中の方位イベントは無視する。


5. ドット表示
--------------------------------------------------

目的地が複数あるため、すべてを見せるか、絞って見せるかが重要です。

おすすめは、未訪問地点のうち、近い順に最大3〜5個だけ表示する方式です。

すべて出すと、都市部や狭い範囲ではドットが多すぎて迷います。
順番を問わない体験なら、近くの反応を選んで向かえる方が自然です。

状態ごとの表示:

  未訪問
    ドット表示対象にする。

  訪問済み
    レーダーから消す。

  到達圏内
    チェックインボタンを出す。

  QR専用地点
    レーダー表示は任意。基本は現地QRで処理する。

ドットには、地点名を出さない方がよいです。
代わりに「反応A」「反応B」「貝の反応」「環境の反応」くらいの抽象表示にします。

ただし、目的地の概要を知らせない方針なら、カテゴリ名も出しすぎない方がよいです。
色や記号だけでも十分です。


6. 距離表示
--------------------------------------------------

「おおよその距離が分かる」ことは重要ですが、数値で細かく出すより、段階表示が向いています。

正確な距離を出すと、GPS誤差で不満が出ます。
また、目的地を伏せたい体験とも少し相性が悪いです。

おすすめの4段階:

  遠い
    表示例: 反応あり
    意味: 方向だけ分かる。

  近い
    表示例: 反応が強い
    意味: だいぶ近い。

  かなり近い
    表示例: 目前に反応
    意味: 周囲を見る段階。

  到達圏内
    表示例: チェックイン可能
    意味: ボタンを表示する。

数値を出す場合も、約300m、約100m、約50m以内のように丸めます。
37mのような細かい表示は、GPS誤差で信用を失いやすいため避けます。


7. チェックイン方式
--------------------------------------------------

地点ごとにチェックイン方式を分けます。

7-1. GPS範囲チェックイン

広場、道中、景観、昔の地形を感じる地点に向いています。

条件:

  ・指定地点の半径内に入る。
  ・チェックインボタンが表示される。
  ・利用者が押して確定する。

自動チェックインは避けます。
利用者が「訪れた」と理解して押す方が、体験としても記録としても自然です。

半径の目安:

  ピンポイント寄り
    30〜50m

  道中の見どころ
    50〜80m

  広場・景観範囲
    80〜120m


7-2. QRチェックイン

案内板、解説板、屋内展示、博物館内に使います。
QRは、その場所に実際に来た証明として扱います。

向いている地点:

  ・解説板の前
  ・案内板の前
  ・博物館内
  ・GPSが弱い場所
  ・建物内
  ・正確にその場所へ立ってほしい地点


7-3. 併用チェックイン

広い範囲ではGPSで「到達」、案内板ではQRで「確定」にする方法です。

例:

  1. 史跡広場の近くに入る。
  2. 「史跡エリアに反応があります」と表示する。
  3. 案内板のQRを読む。
  4. 詳細解説が開く。

この方式が一番確実です。


8. 目的地の種類
--------------------------------------------------

地点は、機能で分けると整理しやすくなります。

  解説地点
    目的: その場所の意味を伝える。
    チェックイン後: 解説、画像、アイテムを表示する。

  道しるべ地点
    目的: 次へ進ませる。
    チェックイン後: 短い反応、次の探知案内を表示する。

  収集地点
    目的: アイテム獲得。
    チェックイン後: カード、素材、達成状況を表示する。

  QR地点
    目的: 実物と結びつける。
    チェックイン後: 現地解説、詳細ページを表示する。

  最終地点
    目的: 達成感を出す。
    チェックイン後: 達成証、認定証案内を表示する。

「道しるべ地点」は説明を短くしてよいです。
すべての地点を濃い解説にすると、体験が重くなります。


9. チェックイン後の表示
--------------------------------------------------

チェックイン後に初めて、その地点の意味を出します。

推奨構成:

  1. 地点名
  2. 獲得アイテム、または達成表示
  3. その地点の意味を短く説明
  4. 詳しく見るボタン
  5. レーダーに戻るボタン

最初に長文を出すとテンポが落ちます。
短文で達成感を出して、詳しい解説は任意にします。

表示例:

  反応を記録しました
  「干潟の記録」を獲得
  この付近は、縄文時代の海辺環境を考える手がかりになる地点です。
  詳しく見る


10. 訪問済み地点
--------------------------------------------------

一度訪れた場所へ再び行く必要はないため、訪問済み地点はレーダーから消します。
ただし、履歴では見返せるようにします。

履歴で見られるもの:

  ・訪問日時
  ・地点名
  ・解説
  ・獲得アイテム
  ・カテゴリ達成状況
  ・最終達成証


11. 達成設計
--------------------------------------------------

動機づけのために、カテゴリ達成と最終達成を入れます。

ただし、目的地の概要を事前に知らせないなら、カテゴリ名も最初は伏せた方が面白くなります。

チェックイン前の表示例:

  ・反応群A
  ・反応群B
  ・反応群C

チェックイン後に開く意味の例:

  ・環境の記録
  ・貝づくりの記録
  ・くらしの記録
  ・交換の記録

最終的には、すべての記録がそろうと達成証が表示される形がよいです。
博物館で紙の認定証と引き換える仕組みとも相性が良いです。


12. 安全面の仕様
--------------------------------------------------

この体験では安全面がかなり重要です。

必須仕様:

  ・探知ボタンを押す前に「立ち止まって周囲を確認」と表示する。
  ・ドットは10秒で消える。
  ・常時ナビにしない。
  ・目的地付近では「周囲を確認してください」と表示する。
  ・車道、線路沿い、私有地付近には地点を置かない。
  ・横断歩道のない道路を渡らせる配置にしない。
  ・到達判定は広めにする。
  ・屋内やピンポイントはQRにする。

これは提案でも強いポイントになります。
楽しさだけでなく、事故を起こしにくい体験設計として説明できます。


13. 技術構成
--------------------------------------------------

Webコンテンツなら、最小構成は以下で十分です。

  現在地取得
    Geolocation API

  方位取得
    DeviceOrientation API

  QRチェックイン
    URLパラメータ方式、またはカメラ読み取り

  訪問履歴
    localStorage

  地点データ
    CSVまたはJSON

  公開
    静的サイト

  サーバーDB
    原則不要

個人情報は扱わない方がよいです。
端末内保存にすれば、運用が軽くなります。


14. 地点データの設計
--------------------------------------------------

CSVよりJSONの方が拡張しやすいです。
ただし、自治体や運用者がExcelで編集するならCSVでもよいです。

最低限必要な項目:

  id
    地点ID

  name
    チェックイン後に表示する地点名

  hiddenLabel
    チェックイン前の表示名。例: 反応A

  lat
    緯度

  lng
    経度

  radius
    GPS判定半径

  checkinType
    gps / qr / hybrid

  group
    達成カテゴリ

  pointType
    explanation / waypoint / collection / final

  itemId
    獲得アイテム

  shortText
    チェックイン直後の短文

  detailText
    詳細解説

  qrId
    QR用ID

  priority
    表示優先度

  enabled
    表示するか

「地点の概要は知らせない」方針なので、hiddenLabel が重要です。
未訪問時は name ではなく hiddenLabel を使います。


15. 表示ルール
--------------------------------------------------

15-1. 探知前

  ・ドットなし。
  ・中央に案内文を表示する。
  ・「探知」ボタンを強調する。

表示文:

  立ち止まって探知すると、周辺の反応を10秒間表示します


15-2. 探知中

  ・未訪問地点の近いものを最大3〜5個表示する。
  ・距離は段階表示にする。
  ・地点名は出さない。
  ・残り5秒で薄くなる。


15-3. 到達圏内

  ・チェックインボタンを表示する。
  ・目的地名は、まだ出してもよい。
  ・完全に伏せたい場合、ボタン名は「反応を記録」にする。

おすすめのボタン名:

  反応を記録


15-4. チェックイン後

  ・地点名を公開する。
  ・解説を表示する。
  ・アイテム獲得を表示する。
  ・履歴に保存する。
  ・レーダーから消す。


16. QRの方式
--------------------------------------------------

QRは2種類あります。

16-1. URL直リンク方式

QRに以下のようなURLを入れます。

  https://example.jp/rally/?qr=POINT_ID

読み取ると、その地点をチェックインします。
これは一番簡単です。
スマホ標準カメラからでも開けます。


16-2. アプリ内カメラ読み取り方式

Web内にQR読み取り機能をつける方式です。

ただし、ブラウザ差や権限問題があります。
安定運用を優先するなら、URL直リンク方式を基本にして、アプリ内読み取りは補助にする方が堅いです。


17. 画面遷移
--------------------------------------------------

画面は以下だけでよいです。

  1. トップ、開始画面
  2. レーダー画面
  3. チェックイン完了画面
  4. 詳細解説画面
  5. 履歴、図鑑画面
  6. 達成証画面
  7. QRエラー、GPSエラー画面

多機能にしすぎると操作が重くなります。


18. エラー時の表示
--------------------------------------------------

GPSや方位は必ず失敗します。
失敗時の文言を最初から設計しておくべきです。

  GPS拒否
    位置情報を許可すると探知できます。QRチェックインは利用できます。

  GPS不安定
    現在地が不安定です。開けた場所で再度探知してください。

  方位不安定
    方角が不安定です。端末を8の字に動かしてから再度探知してください。

  屋内
    屋内ではQRコードを読み取ってください。

  到達圏外
    まだ反応に届いていません。

  訪問済み
    この反応は記録済みです。


19. 実装の優先順位
--------------------------------------------------

第1段階:

  ・地点データ読み込み
  ・GPS取得
  ・距離計算
  ・探知ボタン
  ・方位固定
  ・ドット10秒表示
  ・到達圏内チェックイン
  ・訪問履歴保存

第2段階:

  ・QRチェックイン
  ・チェックイン後の解説
  ・履歴画面
  ・訪問済み除外
  ・エラー表示

第3段階:

  ・アイテム獲得
  ・カテゴリ達成
  ・最終達成証
  ・認定証引き換え案内
  ・演出強化

この順で作ると破綻しにくいです。


20. まとめ仕様
--------------------------------------------------

この体験に最適な仕様:

  体験形式
    探知型ポイントラリー

  目的地表示
    未訪問時は非公開

  方角表示
    探知ボタン押下時のみ

  方位
    探知時点で固定

  ドット表示
    10秒表示、残り5秒でフェード

  距離表示
    数値ではなく段階表示

  訪問順
    自由

  訪問済み地点
    レーダーから除外

  GPS地点
    おおよその範囲でチェックイン

  QR地点
    案内板、解説板、屋内に配置

  保存
    localStorage

  サーバー
    原則不要

  安全対策
    常時ナビにせず、立ち止まって探知する


21. 今後確認したい条件
--------------------------------------------------

次の5点が決まると、仕様をかなり具体化できます。

  1. 目的地の数は、全体で何地点くらいを想定するか。
  2. 1回の体験時間は、15分、30分、60分のどれに近いか。
  3. 博物館から史跡までの道中も対象にするか。それとも史跡周辺だけにするか。
  4. 未訪問地点は完全に伏せるか。それとも「反応の種類」だけは見せるか。
  5. 最終達成後に、博物館で紙の認定証へつなげるか。
